# RP2354B I/O coprocessor

The carrier is a **two-brain** design. The **CM5 drives only the HUB75 face**
(plus cameras, HDMI, USB, power); a dedicated **RP2354B microcontroller owns
every other peripheral** — WS2812 LEDs, the I²C sensor bus, buttons/boop,
the MAX7219 face backend, and **servos** — and talks to the CM5 as a
**USB-CDC device** through the onboard hub.

This extends the existing optional button coprocessor
([`../../docs/coprocessor-input.md`](../../docs/coprocessor-input.md),
`src/input/coproc_inputs.*`, `firmware/button_coproc/`) from "just buttons" to
"all peripheral I/O".

## Why split the brains

| Benefit | How |
|---------|-----|
| **Frees the CM5 GPIO bank** | HUB75 eats 14 of ~28 RP1 GPIO and contends with SPI1; moving everything else off the CM5 ends the pin-contention puzzle entirely (see old `PINMAP.md` free-pin ledger). |
| **HUB75 + MAX7219 can run at once** | They're on different brains now — no shared GPIO/SPI contention. See [`MULTI-BACKEND.md`](MULTI-BACKEND.md). |
| **Hard-real-time I/O** | RP2350 PIO generates WS2812 timing and clean servo PWM; debounce/long-press happen on the MCU, not a Linux poll loop. |
| **Less EMI into the Pi** | Long switch/LED/servo harnesses terminate at the MCU, not the CM5. |
| **Hot-pluggable & re-flashable** | USB-CDC device; reprogram in place (BOOTSEL/UF2) or via the standalone port. |

## Why RP2354B specifically

- **48 GPIO (QFN-80)** — room for sensors + 4 LED zones + 8 servos + 10 buttons
  + SPI + spares with headroom (~30 used of 48).
- **2 MB on-package stacked flash** — no external QSPI flash chip to place/route.
- **PIO** — WS2812 and tidy multi-channel servo PWM without bit-banging the CPU.
- **Dual Cortex-M33 @ 150 MHz** — plenty for I²C polling + USB + PWM + LED frames.

## Pin allocation (source of truth)

RP2354B GPIO are **3.3 V**, not 5 V-tolerant — same rule as the CM5. 5 V-logic
loads (WS2812, MAX7219) are level-shifted on the **RP2354B side**; I²C sensors
and buttons are 3.3 V-native and connect directly.

| GPIO | Function | Net | Shift | Notes |
|-----:|----------|-----|-------|-------|
| GP0 | UART0 TX | `DBG_TX` | — | debug console header |
| GP1 | UART0 RX | `DBG_RX` | — | debug console header |
| GP2 | SPI0 SCK | `MX_CLK` | →5 V | MAX7219 clock |
| GP3 | SPI0 TX  | `MX_DIN` | →5 V | MAX7219 data-in |
| GP4 | I²C0 SDA | `SDA0` | — | sensor bus + 4.7 k pull-up |
| GP5 | I²C0 SCL | `SCL0` | — | sensor bus + 4.7 k pull-up |
| GP6 | input | `SENS_INT` | — | MPR121/IMU interrupt |
| GP7–GP10 | SPI0 CS×4 | `MX_CS1..4` | →5 V | MAX7219 chains 1–4 |
| GP11–GP15 | spare | — | — | UART1 / 2nd I²C / extra CS |
| GP16–GP19 | PIO WS2812 | `LED1..4_DAT` | →5 V | 4 addressable LED zones |
| GP20–GP27 | PWM | `SRV1..8` | 3.3 V | **8 servo channels** (direct) |
| GP28–GP37 | input + pull-up | `BTN1..10` | — | buttons / boop (MCU debounce) |
| GP38–GP39 | spare digital | — | — | expansion |
| GP40–GP47 | ADC0–7 | `AIN0..7` | — | analog: flex sensors, pots, batt sense |
| USB DP/DM | native USB | `RP_DP/DM` | — | → USB selector (§USB) |
| XIN/XOUT | 12 MHz XTAL | — | — | required for USB |
| SWCLK/SWDIO | SWD | — | — | debug/flash header |

> Servos: 8 GPIO on the MCU, but **8× 3-pin headers (24 positions)** on the
> board — V+ and GND are the shared servo rail, only the 8 signals are unique.

## Level shifting on the RP2354B side

| Load | Logic | Shifter | Part |
|------|-------|---------|------|
| MAX7219 (DIN/CLK/CS×4) | 5 V | 3.3→5 V octal buffer | 74AHCT245 |
| WS2812 (4 data lines) | 5 V | 3.3→5 V single/quad buffer | 74AHCT125 |
| Servos (8 signals) | accepts 3.3 V | **none** (direct) | — |
| I²C sensors | 3.3 V | none | — |
| Buttons | 3.3 V | none | — |

`74AHCT*` (TTL VIH ≈ 2 V) reads the RP2354B's 3.3 V as logic-high while driving
clean 5 V — exactly the HUB75 buffer rationale, applied to the MCU's loads.

## CM5 ↔ RP2354B link — USB CDC

The RP2354B enumerates as a **USB CDC ACM** device (`/dev/ttyACM*`) behind the
onboard USB hub. The wire protocol extends the v1 ASCII line protocol in
[`../../docs/coprocessor-input.md`](../../docs/coprocessor-input.md):

- **MCU → CM5:** `HELLO proto-io v1 …`, button events (`BTN <id> SHORT|LONG`),
  sensor frames (IMU quat/accel, boop channels, lux), ADC values.
- **CM5 → MCU:** LED frames/effects, servo targets (`SRV <id> <us>`), MAX7219
  pixel data, config (button maps, pull bias, I²C addresses).

The CM5 firmware dispatches button events through the **same `gpio_dispatch`
path** as on-board GPIO, so downstream menu/boop/camera code is unchanged.

## Programming paths (two)

1. **In-system (normal):** through the CM5-hub USB link — `picotool`/UF2 over
   USB, BOOTSEL button to enter the bootloader. The CM5 can re-flash the MCU.
2. **Standalone (isolated):** a dedicated **USB-C port** on the RP2354B's single
   USB pair, selected by a **switch/mux** that disconnects it from the CM5 hub
   (see USB selector below). Flip to standalone → plug into any computer →
   flash, with the CM5 fully out of the loop.

Plus an **SWD header** (SWCLK/SWDIO/GND/3V3) for debug-probe flashing and live
debug, and a **RUN/reset** button.

## USB selector (CM5 hub ⇄ standalone port)

The RP2350 has **one** USB controller (one D+/D− pair), so the two paths are
mutually exclusive and share that pair through a selector:

```
RP2354B USB ──► [ USB 2.0 selector ] ──► A: onboard hub downstream → CM5
   (DP/DM)         (SW1 / TS3USB221)      B: standalone USB-C port (program)
```

- **SW1** = a DPDT slide switch on D+/D− (fine for RP2350's full-speed USB) **or**
  a `TS3USB221A` mux for cleaner switching.
- **VBUS isolation:** the RP2354B is powered from the carrier's 3V3 rail, so the
  standalone port's VBUS is used only for host-present detection — do **not**
  back-feed it onto the carrier or the hub. Steer/diode-OR or leave VBUS sense
  only.

## Power

| Rail | Source | Feeds |
|------|--------|-------|
| `+3V3_RP` | small LDO/buck off `+5V` (≥ 500 mA) | RP2354B core/IO + I²C sensors + buffers' A-side |
| `+5V` | main 5 V bus | MAX7219/WS2812 buffer B-side (VCC) |
| `+V_SERVO` (5–6 V) | fused tap off `+5V` (or dedicated 6 V buck) | servo V+ rail only |

Sensors now draw from **`+3V3_RP`** (not the CM5 3V3). Servo rail is its own
fused, bulk-capped domain so stall surges don't disturb logic — see
[`POWER.md`](POWER.md).

## Firmware

Lives at `firmware/io_coproc/` (RP2350 SDK / PlatformIO). Cores:
PIO for WS2812 + servo PWM; I²C polling of the sensor set; USB-CDC protocol;
button debounce. The existing `firmware/button_coproc/` is the seed.
See [`../../docs/coprocessor-input.md`](../../docs/coprocessor-input.md).
